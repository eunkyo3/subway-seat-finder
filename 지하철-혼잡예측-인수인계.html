1. 프로젝트 한눈에 보기

기존 지하철 앱은 “열차가 몇 분 뒤 온다”는 현재 상태만 보여준다. 이 프로젝트는 한 발 더 나아가 “앞으로 얼마나 붐빌지”를 예측하고, “이번 열차를 탈까 / 한 대 보낼까 / 몇 정거장 뒤 앉을 수 있을까”라는 행동을 추천한다.

💡 한 줄 정의

서울 지하철 1~8호선 대상으로, 실시간 열차 위치를 지도에 띄우고 + 이번 열차의 예상 혼잡도를 계산해 “어느 열차를 타야 덜 붐비는지”를 추천하는 로컬 Docker 라이브 웹 대시보드.

본 프로젝트는 두 선행 프로젝트(NutriRank = 읽기전용 앱 + 배치 파이프라인 / weather-crash-risk = 지도 시각화 + 한계 정직 문서화)의 구조를 재사용한다.

2. 무엇을 만드나 (핵심 기능)

① 실시간 노선도

1~9호선+광역철도 열차가 노선도 위에서 실제 위치대로 움직인다.

② 이번 열차 예상 혼잡도

도착 예정 열차의 혼잡도를 %·등급(여유/보통/혼잡)으로 표시.

③ 이번 vs 다음 추천

배차간격 기반으로 “다음 열차가 더 여유로워요” 추천. 차이가 미미하면 “비슷함”.

④ 앉을 타이밍 예측

“3정거장 뒤 사당에서 대량 하차 → 그때부터 앉을 확률↑” 타임라인.

⑤ 빈 열차(시발) 알림

중간역 시발로 텅 비어 오는 열차를 감지해 우선 추천.

⑥ 이동량 분석

역별·시간대별 승하차로 혼잡 히트맵과 이동 패턴 시각화.

3. 아키텍처 · 동작 방식 · 시스템 구조도

데이터를 성격에 따라 두 갈래로 나눠 다룬다. 하나는 미리 준비해 DB에 넣어두는 배치 데이터, 다른 하나는 데모 순간 불러오는 실시간 위치다.

  • ① 배치 데이터 (혼잡도·승하차 등) = 다른 곳에서 준비 → DB에 적재. 혼잡도·승하차는 실시간 데이터가 없고 과거 통계뿐이므로, 별도로 준비·가공한 데이터를 DB에 “갈아끼거나 추가”한다. 예측 계산의 기준값이 된다.
  • ② 실시간 열차 위치·도착 = 데모 순간 라이브 API 호출 → 지도의 열차가 “진짜 지금” 위치대로 움직인다.
  • 데모 = ①을 넣어둔 상태 + ②를 라이브로 표시. 즉 미리 적재한 배치 데이터 위에서, 실시간 위치를 얹어 예측·추천을 보여준다.

🔁 “데이터 갈아끼기”의 정확한 의미

앱이 실시간 위치를 미리 녹화해 두는 게 아니다. 배치로 준비한 혼잡도·승하차 데이터를 DB에 넣거나 최신 것으로 교체(갈아끼기)·추가하는 데이터 관리/갱신을 뜻한다. 앱은 그 DB를 읽고, 위치만 라이브로 호출한다.

[ 배치 준비 (앱과 분리) ] 혼잡도·승하차 등 데이터 가공 → DB에 적재 / 갈아끼기 / 추가
    ↓
[ 로컬 Docker 웹앱 실행 ] DB의 배치 데이터 로드
    + 실시간 위치·도착 API 라이브 호출 (데모 순간)
    ↓
[ 예측 엔진 ] 기준혼잡도(배치) + 배차간격·시발감지(실시간 위치) + 하차궤적(배치) → 예상혼잡도
    ↓
[ 화면 ] 노선도 + 실시간 열차 마커 + 이번/다음 카드 + 히트맵

[ (선택) 안전망 ] 네트워크·API 장애 대비, 미리 저장한 위치 스냅샷 재생

✅ 왜 로컬 Docker인가

24시간 상시 서버·클라우드 배포가 불필요하다. 혼잡도·승하차 배치 데이터는 앱과 분리해 따로 준비·적재하고, 앱은 그 DB를 읽으며 실시간 위치만 라이브로 호출한다. docker compose up 한 번으로 어디서든 재현 가능 — NutriRank의 “읽기전용 앱 + 별도 배치” 구조와 동일.

⚠️ 데이터 성격 주의

실시간으로 받을 수 있는 건 위치·도착뿐이다. 혼잡도·승하차는 과거 통계·집계이며 실시간 값이 없다. 따라서 “예측 계산”은 구조적으로 배치 데이터 기반일 수밖에 없다.

5. 지원 호선 & subwayId 코드표

실시간 위치·도착 API는 1~9호선을 넘어 광역철도까지 지원한다. 다만 혼잡도 통계는 서울교통공사 운영 1~8호선만 존재한다. 그래서 2단계 구현을 한다.

6. API 신청 방법 (단계별)

신청은 서울 열린데이터광장 한 곳이면 끝난다. (SKT/공공데이터포털 불필요)

1

회원가입 & 로그인

data.seoul.go.kr 접속 → 회원가입.

2

인증키 발급

마이페이지 → 인증키 신청·관리에서 일반 인증키 1개 발급. 이 키 하나로 실시간 위치·도착 API를 모두 호출한다. 대체로 즉시 발급된다.

3

혼잡도·승하차 파일 다운로드

OA-12928(혼잡도), OA-12921(승하차)은 파일 다운로드라 인증키 없이 로그인만으로 내려받는다.

4

(권장) 활용사례 갤러리 등록 → 호출 제약 해제

서울 광장은 인증키에 기본 호출 제한이 있으나, 활용사례(갤러리)에 프로젝트를 등록하면 “이용 제약 없이” 사용할 수 있다. 실시간 수집을 하려면 등록해두는 것이 안전하다. (Open API 이용가이드)

✅ 호출 예시 (실시간 도착)

GET http://swopenapi.seoul.go.kr/api/subway/{인증키}/json/realtimeStationArrival/0/10/{역명}

# 실시간 위치 (호선 단위)
GET http://swopenapi.seoul.go.kr/api/subway/{인증키}/json/realtimePosition/0/100/{호선명}

응답은 JSON. {인증키}를 발급받은 키로 바꾸면 된다. 정확한 엔드포인트/파라미터는 각 데이터셋 명세서에서 최종 확인.

8. 예측 알고리즘

핵심은 “이번 vs 다음”을 무엇으로 가르느냐이다. 이번·다음 열차는 몇 분 차이라 같은 30분 통계 구간에 들어가 통계만으로는 구분되지 않는다. 그래서 실시간 신호 3가지로 보정한다.

🧮 예측 공식

예상혼잡도 = 기준혼잡도(역·요일·시간대 통계)
           × 배차간격 보정   ← 앞 열차와 간격이 길수록 승객 누적↑ = 더 붐빔
           × 시발 보정       ← 시발역 출발 빈 열차면 하향
           − 하차궤적 보정   ← 내 목적지까지 대량 하차역에서 감소

추천 = 이번/다음 중 min(예상혼잡도)
       단, |이번 − 다음| < 임계값 이면 "비슷함"으로 표시(과잉추천 방지)
신호계산 방법데이터 출처
배차간격(headway)실시간 도착 간격 또는 열차번호 궤적으로 앞 열차와의 시간차 산출. 이번/다음을 가르는 핵심 지표OA-12601 / OA-12764
시발 감지열차번호(btrainNo)가 처음 등장한 역을 추적 → 중간 시발이면 빈 열차. (회차/입고 열차와 구분 신뢰도는 수집 데이터로 검증)OA-12601 (수집 축적)
하차 궤적역별 혼잡도 궤적 + 승하차로 “어느 역에서 대거 빠지는지” 계산 → 목적지 전까지 앉을 타이밍 추정OA-12928 + OA-12921

배차간격 보정계수는 초기엔 정성적 규칙으로 두고, 수집 데이터가 쌓이면 캘리브레이션(튜닝)한다. 후속으로 ML 모델(LightGBM 등)로 업그레이드 여지.

9. 검증 방법

⚠️ 핵심 약점 = 실측 정답 부재

열차 내 재차인원 센서가 없어 “정확히 몇 명”의 실측 정답이 없다. 그래서 인원수(명)가 아니라 혼잡도 %(추정)로 표현한다.

검증 전략(교수의 “그게 맞는지 어떻게 아냐” 방어용):

  • 공식 통계를 의사정답(pseudo-ground-truth)으로: 예측 혼잡도를 공식 30분 평균 혼잡도(OA-12928)와 대조해 예측 오차를 측정
  • 배차간격–혼잡 상관분석: 우리가 수집한 실시간 데이터에서 “간격이 길수록 붐빈다” 가설을 통계로 검증
  • 한계 정직 명시: 표본·센서 한계를 문서화(weather-crash-risk 스타일) → 오히려 학술적 신뢰 확보

10. 기존 앱 대비 차별점

항목기존 앱 (카카오맵·또타지하철·T맵 등)본 프로젝트
혼잡 정보현재 역 혼잡도(상태)열차별 예측 + 노선 따라 변하는 궤적
이번/다음도착시간만 표시배차간격 기반 “뭘 타라” 추천
앉을 자리칸별은 일부(2·3호선)만“몇 정거장 뒤 앉는다” 타이밍 예측
빈 열차안 알려줌시발 빈 열차 우선 추천
데이터 활용실시간만 소비실시간을 축적 → 패턴·예측·분석

🎯 한 줄 가치 제안

“도착 정보는 흔하지만, 배차간격·하차궤적·시발열차를 종합해 ‘어느 열차를 타야 앉는지’를 예측·추천하는 서비스는 없다.” — 상태 표시(정보) → 행동 추천(의사결정)으로 한 단계 올린다.

-- 실시간 위치 로그 (수집기가 주기적으로 축적)
train_position_log(

arrival_log(

)

-- 혼잡도 통계 (정적, 파일 적재)
congestion_stat(
  subway_id, station_id, day_type,      -- 평일/토/일
  time_slot,                            -- 30분 단위
  congestion_pct
)

station_flow(
  station_id, date, time_slot, board_cnt, alight_cnt

station_master(

핵심 파생값: (1) 열차번호별 궤적 → 시발 감지 · 배차간격, (2) congestion_stat × station_flow → 하차궤적, (3) 최근 스냅샷 저장 → 재생 폴백.

메인: 실시간 노선도

역 선택 패널

역 클릭 시 “이번 열차 카드”: 도착 O분 · 예상 혼잡도 %·등급 · 급행 여부.

이번 vs 다음 카드

두 열차 비교 + “다음이 더 여유로워요” 추천 배지.

앉을 타이밍 타임라인

목적지까지 정거장별 예상 혼잡 변화 + “여기서부터 앉을 확률↑” 마커.

혼잡 히트맵

역별·시간대별 평균 혼잡을 색으로. “언제 타면 앉나” 인사이트.

재생 컨트롤

API 실패 시/데모용으로 수집 스냅샷을 시간축 따라 재생.

13. 남은 확인 항목 & 리스크

아래는 구현을 막지는 않지만 착수 전/초기에 확정해야 하는 항목이다.

  • 서울 광장 API 정확한 일일 호출 한도 확인 (데이터셋 API 상세) + 갤러리 등록 여부 결정
  • 실시간 위치·도착 API의 최신 지원 노선·subwayId 코드 명세 확인
  • 배차간격 보정계수 캘리브레이션 방법 (수집 데이터로 튜닝)
  • 배치 데이터(혼잡도·승하차)의 기간·최신성 및 평일/주말/공휴일 커버리지 확인 + 갱신(갈아끼기) 주기 결정
  • 지연·사고일 이상치 처리 규칙 (배차간격 급변 제외 여부)
  • 노선도 좌표/SVG 확보 (역 좌표 데이터 매핑)
  • 시발 감지 신뢰도 검증 (회차/입고 열차와 구분)

알려진 한계 (정직하게 명시할 것)

• 정확한 인원수(명) 불가 → 혼잡도 %로 대체
• 예측은 “특정 열차 실측”이 아닌 “시간대 통계 기대값 + 실시간 보정”
• 하차궤적·혼잡은 역·시간대 평균 기반이라 개인화 정밀도에 본질적 한계
• 9호선·광역철도는 혼잡도 통계 없음 → 위치·도착만

14. 발표 포인트

  • 빅데이터 서사: “역별·시간대별 혼잡도·승하차 등 대량의 배치 데이터를 가공·적재하고, 여기에 실시간 열차 위치를 결합해 예측했다. 데이터는 배치로 계속 갈아끼우고 추가하며 갱신된다.” → 데이터 엔지니어링(배치 파이프라인) 역량 어필
  • 차별점 방어: “SKT/또타지하철도 혼잡도를 보여주지 않나?” → “그들은 현재 상태만, 우리는 예측 + 이번/다음 추천. 상용 서비스가 아니라 공공데이터로 수집→모델→시각화 파이프라인을 구축한 PBL이다.”
  • 정직한 한계: 실측 부재를 숨기지 않고 검증 방법(공식 통계 대조)과 함께 제시 → 학술적 신뢰
  • 시연 안전망: 재생 폴백 덕분에 발표 중 네트워크·API 문제에도 시연이 끊기지 않는다.
